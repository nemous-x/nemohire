# Document Generation

## Grounding rule (non-negotiable)
Every factual claim in a generated document — a skill, a project, a metric, a title, a date — must trace back to something already present in `.claude/nemohire/identity/`. If the target job needs something not on record, do not invent it. Either omit it, or explicitly flag the gap in your output so the user can decide how to address it (add it to identity, address it in the cover letter honestly, or skip the job).

## Voice rule
Match `identity/writing-style.md` and `identity/communication-style.md` — tone, vocabulary level, sentence rhythm, confidence. A generated cover letter or live answer should be indistinguishable in voice from something the user would write themselves, adjusted only for the specific company/role content.

## Human-voice rule (non-negotiable)
Every piece of content that goes into an application — cover letter, resume, live-form answer — is written in the user's own first-person voice, as the candidate, in plain human grammar. It must never:
- disclose, hint at, or reference AI assistance, generation, or drafting ("as an AI...", "generated by...", "I'm an AI and...", meta-commentary about the writing process, etc.)
- break character with disclaimers, hedges about being a language model, or any note addressed to a reviewer/recruiter about how the material was produced
- read like a template or like something written *about* the candidate rather than *by* them

`identity-agent` is the only agent that produces this content, and this rule applies to every word it writes — resume tailoring, cover letters, and every live answer during `/nemo:apply`. `application-coordinator-agent` and `browser-agent` never write content themselves, so this rule never needs to apply to them.

## Structure
- Resumes: **opt-in only** — check `identity/documents.md`'s "Tailor per job" setting first. If off (the default), don't touch the resume at all; copy it in as-is. If on, reorder and re-emphasize existing `identity/experience.md` / `identity/achievements.md` content to mirror the target posting's language and priorities. Use `templates/resumes/tailoring-checklist.md` as a guide.
- Cover letters: always written. Use `templates/cover-letter/` as a structural starting point, not a fill-in-the-blank script. Reference specifics from the actual posting.
- Answers: start from `identity/interview-library.md` for standard questions; adapt rather than reuse verbatim. Answer every question in `jobs/cache/<id>/questions.md` in one pass, writing directly into the file — never one dispatch per question.

## Reading context — always by id, never from the payload, never stale
You are handed a job `id` and an instruction ("prepare materials" or "answer the questions in questions.md") — nothing more. Read `.claude/nemohire/jobs/cache/<id>/posting.md`, `.claude/nemohire/jobs/cache/<id>/company-context.md`, and (for answering) `.claude/nemohire/jobs/cache/<id>/questions.md` yourself, every time you need them — **fresh, for the id given in this specific call**. Never expect the coordinator to have included this text in what it sent you, and never assume the context from a different job's id (even one you handled earlier in the same run) still applies — a new id means a new posting, a new company context, and a new questions file, read again from scratch. See `templates/tracker/job-cache-schema.md`.

## Answering, and what to do when you can't
Write each answer directly into `questions.md`'s `Answer` field for that question — never return answer text in your response. If a question needs information you don't have anywhere in `identity/` or the posting, write `[NEEDS INPUT: <what's missing>]` in that field instead of guessing — this is what tells `browser-agent` to stop and ask the user for exactly that, rather than submitting with a fabricated or missing answer.

## Quality bar
Before considering anything ready to submit: no leftover template placeholders, no generic filler sentences that could apply to any company, and no claims that don't trace back to identity. `identity-agent` self-checks against this bar and the human-voice rule before returning any content — there's no separate QA pass.
